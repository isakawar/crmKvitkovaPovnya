# Business Functions Analysis — CRM Kvitkova Povnya

> Methodology:
> - **Observed behavior** — що робить код зараз (факт)
> - **Intended behavior** — гіпотеза про те, що мало б робити (позначено як *hypothesis*)
> - **Suspicious** — підозрілі місця, потенційні баги, неочевидна логіка

---

## 1. Управління клієнтами

### Observed behavior
- Клієнт ідентифікується виключно по `instagram` (унікальний рядок, обов'язковий).
- При створенні замовлення — пошук клієнта тільки по Instagram; якщо не знайдено — помилка (не автоствоення).
- Додаткові поля: `phone`, `telegram`, `credits`, `marketing_source`, `personal_discount`.
- CRUD доступний через `/clients`.

### Intended behavior (*hypothesis*)
- Instagram є основним каналом комунікації з клієнтами магазину. Облік клієнта без Instagram неможливий або небажаний.
- `personal_discount` і `credits` — інструменти утримання / програми лояльності.

### Suspicious
- **`personal_discount` — рядок (`String(16)`)**: зберігається як текст, не число. Незрозуміло, чи він застосовується автоматично до замовлень чи є лише довідковою інформацією. У моделі Order жодних посилань на нього немає.
- **`credits` оновлюється при поповненні** (через `Transaction` credit), але **не зменшується при списанні** через write-off транзакцію. Тобто `client.credits` і сума дебетових транзакцій клієнта — різні речі.

## 2. Разове замовлення (One-time order)

### Observed behavior
- Створює `Order` + рівно одну `Delivery` з однаковими полями (адреса, час, розмір).
- Адреса денормалізована: зберігається і в `Order`, і в `Delivery`.
- При `is_pickup=True` — `street` в Delivery = `None`, в Order = `'Самовивіз'`.
- Додаткові телефони (`RecipientPhone`) зберігаються окремо.
- При редагуванні (`update_order`): всі активні доставки отримують оновлені поля адреси/коментаря, але **тільки перша за датою** отримує нову дату доставки.
- При видаленні — каскадно видаляються Delivery та RouteDelivery.

### Intended behavior (*hypothesis*)
- Одне замовлення = один букет, одна дата, одна адреса.
- Замовлення може бути або кур'єрське, або Нова Пошта, або самовивіз.

### Suspicious
- **При редагуванні дата оновлюється тільки для першої доставки** (`active_deliveries[0]`). Для разових замовлень — зазвичай одна доставка, тому ОК. Але якщо доставок більше (напр. через баг або import) — інші залишаться зі старою датою.
- **`delivery_date` є і в Order, і в Delivery**: потенційна розсинхронізація. Canonical-джерело неочевидне.
- **`for_whom` — обов'язкове поле в моделі, але при update**: `order.for_whom = form.get('for_whom') or order.for_whom` — старе значення зберігається якщо форма порожня. В create — `or ''`.

## 3. Підписки (Subscriptions)

### Observed behavior
- Підписка — окрема модель `Subscription` з типом `Weekly` / `Bi-weekly` / `Monthly`.
- При створенні підписки автоматично генерується рівно **4 Orders + 4 Deliveries** на обчислені дати.
- Дати обчислюються від першої дати: кожна наступна — `+7` / `+14` / `+28` днів до цільового дня тижня.
- Час доставки (`time_from` / `time_to`) заповнюється тільки для першого ордера. Коментар — теж тільки для першого.
- `additional_phones` зберігаються тільки для першого ордера в циклі.
- `delivery_day` — день тижня (ПН-НД); якщо не вказано — береться з першої дати.

### Intended behavior (*hypothesis*)
- Клієнт замовляє букети з регулярністю: щотижня / раз на 2 тижні / раз на місяць.
- Цикл = 4 доставки. Після закінчення циклу — пропозиція продовжити.
- `is_wedding` — спеціальний маркер для весільних підписок (можливо, інша логіка ціноутворення або пріоритет).

### Suspicious
- **Monthly = +28 днів, не реальний місяць**: при місячній підписці дата обчислюється як `previous + 28 days`, а потім коригується до потрібного дня тижня. Якщо місяць має 31 день — зсув накопичується. Через 12 місяців дати можуть суттєво "поїхати".
- **Bi-weekly може додати до 20 днів**: `+14 days` + корекція weekday `% 7` вперед. Якщо поточний день = вівторок, а цільовий = понеділок — буде +13 від базової точки, тобто реально `+14+6=20` днів.
- **`is_first` параметр в `_create_delivery_for_order` ніколи не використовується** всередині функції — мертвий параметр.
- **Час/коментар тільки для першої доставки**: це може бути свідомим вибором (час узгоджується окремо), але неочевидно.

## 4. Продовження підписки (Subscription Extension)

### Observed behavior
- `extend_subscription` знаходить останню доставку, обчислює наступну дату після неї, генерує ще 4 ордера/доставки.
- Встановлює `is_extended = True`, `followup_status = 'extended'`, `followup_at = now`.
- Sequence numbers продовжуються від `max_seq + 1`.

### Intended behavior (*hypothesis*)
- Після закінчення циклу менеджер пропонує клієнту продовжити підписку. Якщо клієнт погоджується — запускається продовження.

### Suspicious
- **`followup_status = 'extended'` встановлюється тільки через `extend_subscription`**. Але в `dashboard.update_subscription_followup` той самий статус можна встановити через POST без реального продовження (просто позначка). Тобто `is_extended=True` не гарантує, що були згенеровані нові доставки.
- **Продовження генерує рівно 4 ордери незалежно від типу підписки**: для Weekly — 4 тижні вперед, для Monthly — теж 4, що = 4 місяці. Може бути неочевидним для Monthly.

## 5. Управління доставками

### Observed behavior
- Статуси: `Очікує` → `Розподілено` (призначено кур'єра) → `Доставлено` / `Скасовано`.
- `delivered_at` фіксується при першому переході в `Доставлено`.
- `status_changed_at` оновлюється при кожній зміні статусу.
- `deliveries_count` кур'єра інкрементується **тільки при переході** `Розподілено → Доставлено`.
- Фільтрація доставок: по даті, по клієнту (instagram), по телефону отримувача, по статусу, по фінансовому тижню.
- Фінансовий тиждень = субота–п'ятниця (не стандартний пн-нд).
- Адреса в Delivery денормалізована відносно Order.

### Suspicious
- **`deliveries_count` не інкрементується при прямому переході** `Очікує → Доставлено` (напр. через флориста, або через edit). Тільки перехід саме з `Розподілено`.
- **Фінансовий тиждень субота-п'ятниця**: нестандартно, але можливо специфіка бізнесу (суботній ринок?). Немає пояснення в коді.
- **`update_delivery` при `size == 'Власний'` оновлює `order.custom_amount`**: side effect на батьківський ордер при редагуванні доставки. Потенційна плутанина.

## 6. Флорист-інтерфейс

### Observed behavior
- Окремий стан `florist_status`: `Зібрано` / `Передано кур'єру` / `Доставлено`.
- При `handoff` — якщо delivery ще не `Доставлено`, ставиться `status = 'Розподілено'`.
- При `delivered` через флориста — ставиться `status = 'Доставлено'` і `delivered_at`.
- Флорист бачить доставки на обрану дату, згруповані: маршрутні / позамаршрутні / Нова Пошта / самовивіз.
- Індекс позиції доставки всередині ордера (`subscription_delivery_index`) — для відображення "1-а доставка з 4-х".

### Intended behavior (*hypothesis*)
- Флорист збирає букети, передає кур'єру, або безпосередньо відмічає доставлено (напр. самовивіз).
- Окрема роль — не бачить фінансів, клієнтів тощо.

### Suspicious
- **`florist_status = 'Доставлено'` дублює `status = 'Доставлено'`**: два поля, два джерела правди. При переході через флориста обидва оновлюються. При переході через кур'єра/адміна — `florist_status` не оновлюється.
- **`_build_subscription_delivery_index` рахує позицію по delivery в межах order_id** (не subscription_id): якщо у ордера є кілька доставок — лічильник буде некоректним відносно загального циклу підписки.

## 7. Маршрути (Route Planning)

### Observed behavior
- `DeliveryRoute` = набір зупинок (`RouteDelivery`) для одного кур'єра на одну дату.
- Маршрут можна відправити кур'єру в Telegram (`assign_and_send_route`): надсилається повідомлення з адресами, посиланням на Google Maps, кнопками "Прийняти" / "Відхилити".
- Кур'єр може прийняти або відхилити маршрут через Telegram.
- При зміні кур'єра — старому надсилається повідомлення про скасування.
- Старт маршруту: вручну вказується `start_time`, система обчислює `planned_arrival` для кожної зупинки через `duration_from_previous_min`.
- При видаленні маршруту — доставки повертаються в стан `Очікує`.
- Сторінка також показує: Nova Poshta deliveries, самовивіз, не-маршрутизовані доставки.

### Intended behavior (*hypothesis*)
- Щодня менеджер будує маршрути через оптимізатор (зовнішній API), призначає кур'єра, відправляє в Telegram на затвердження.

### Suspicious
- **`cached_result_json` в DeliveryRoute**: є поле для кешу відповіді від оптимізатора, але `cached_at` — логіка інвалідації кешу не видна в прочитаних файлах.
- **Google Maps URL будується конкатенацією адрес** без нормалізації — якщо адреса містить кирилицю без encoding — URL може бути некоректним (хоча є `urllib.parse.quote`).
- **`status` маршруту**: `draft → sent → accepted/rejected → completed`. Переходу в `completed` в прочитаному коді не знайдено — можливо не реалізовано.

## 8. Telegram-бот для кур'єрів

### Observed behavior
- Реєстрація: кур'єр надсилає `/start`, ділиться номером телефону, система шукає кур'єра по телефону в БД.
- Після реєстрації кур'єр прив'язується до `telegram_chat_id`.
- Кур'єр отримує пропозицію маршруту (inline кнопки: прийняти/відхилити).
- Є команди: `/deliveries`, `/today`, `/tomorrow`, `/week`.

### Intended behavior (*hypothesis*)
- Кур'єри не мають доступу до веб-CRM. Весь їх workflow — через Telegram.
- Маршрут треба явно прийняти до початку виконання.

### Suspicious
- **Реєстрація через телефон**: кур'єр має бути попередньо створений в CRM з правильним номером телефону. Якщо телефон не збігається — реєстрація провалиться без зрозумілої помилки для кур'єра.
- **`telegram_notifications_enabled`** — поле є, але логіка перевірки при відправці сповіщень не видна в handlers.py.

## 9. Сертифікати

### Observed behavior
- Типи: `amount` (сума грн), `size` (розмір букету), `subscription` (підписка + розмір).
- Код автогенерується: `П0001` для підписок, `Р0001` для решти (sequential).
- Термін дії — 1 рік від дати створення.
- Статуси: `active` / `used` / `expired`.
- `effective_status()` — динамічний: якщо `status='active'` але `expires_at < today` — повертає `'expired'`.
- При перевірці (`/certificates/validate`) — перевіряється `status` і `expires_at`, але `status` в БД не оновлюється автоматично до `expired`.
- Прив'язка до замовлення через `order_id` (nullable).

### Intended behavior (*hypothesis*)
- Сертифікат використовується при оформленні замовлення — вводиться код, система перевіряє валідність і визначає параметри замовлення (суму знижки / розмір / тип підписки).

### Suspicious
- **`status` в БД не переходить в `expired` автоматично**: при фільтрації списку в `/certificates` використовується OR-умова `(status == 'expired') | (status == 'active' AND expires_at < today)`. Тобто в БД `expired`-сертифікати залишаються зі статусом `active` — пляшкове горлечко для звітів.
- **Код генерується по `id DESC`**, але при колізії (код вже існує) — робиться `int(existing.code[1:]) + 1`. Якщо в БД є Р0005, Р0003, Р0001 — наступний буде Р0006 (від останнього за ID). Якщо хтось вручну вписав код Р9999 — наступний автоген може дати Р10000 (5 цифр замість 4).
- **`Certificate.is_active()` і `Certificate.effective_status()` роблять схоже**: два окремих методи, частково дублюють логіку.
- **При використанні сертифіката в замовленні** — немає явного переходу `status → 'used'` і запису `used_at` / `order_id` в прочитаних файлах (можливо в order_service при збереженні).

## 10. Транзакції / Фінанси

### Observed behavior
- Два типи: `credit` (дохід від клієнта) і `debit` (витрата магазину).
- `credit`: прив'язаний до клієнта, `payment_type` = `monobank` | `cash`. При записі — `client.credits += amount`.
- `debit`: без прив'язки до клієнта, `expense_type` — текстовий рядок.
- Загальний баланс = сума всіх credits мінус сума всіх debits.
- Місячний дохід = сума credits за поточний місяць.

### Intended behavior (*hypothesis*)
- `client.credits` — баланс клієнта (передоплата або кредит довіри), окремо від загальної каси магазину.
- Транзакції — проста каса без подвійного запису.

### Suspicious
- **`client.credits` і транзакції — не синхронізовані**: дебетова транзакція (`debit`) не зменшує `client.credits`. Тобто `client.credits` = сума всіх **credit** транзакцій клієнта, але не відображає фактичний залишок якщо з нього списували.
- **`debit` транзакції не мають клієнта**: це витрати магазину (зарплата, оренда, квіти), а не списання з клієнта. Два концепти в одній таблиці.
- **Немає прив'язки транзакцій до замовлень**: неможливо зрозуміти, яка транзакція оплачує яке замовлення.

## 11. Звіти / Аналітика

### Observed behavior
- Розподіл замовлень по: маркетингового джерела (all time + поточний місяць), `for_whom`, тип доставки, розмір.
- Тренд замовлень по місяцях (останні 12 місяців): загальна кількість + кількість підписних.
- Нормалізація значень: різні варіанти "не вказано" (`''`, `'-'`, `'—'`, `'none'` тощо) зводяться до `'Не вказано'`.

### Intended behavior (*hypothesis*)
- Аналіз ефективності маркетингових каналів.
- Розуміння для кого купують квіти (собі / подарунок / весілля).
- Моніторинг зростання підписок.

### Suspicious
- **`_monthly_orders_trend` рахує Orders, не Subscriptions за датою замовлення**: підписка з 4-ма ордерами рахується як 4 замовлення в момент створення. Може спотворювати тренд ("сплески" при пакетному імпорті).
- **Місячний фільтр базується на `Order.created_at`**, а не на `delivery_date`: замовлення на грудень, створені в листопаді — потраплять у листопад.

## 12. Dashboard

### Observed behavior
- Показує стан доставок на сьогодні: всього / виконано / в кур'єрів / не призначено.
- Порівняння з "тиждень тому" (той самий день тижня).
- Breakdown по типу: кур'єр / Нова Пошта / самовивіз.
- Нові замовлення / підписки / продовження за сьогодні.
- Тренд доставок за останні 7 днів.
- **Список підписок для follow-up**: підписки, у яких остання доставка була 4+ дні тому, `is_extended=False`, `followup_status` = NULL або `pending`.

### Intended behavior (*hypothesis*)
- Менеджер бачить операційну картину дня.
- "Ended subscriptions" — сигнал зателефонувати клієнту і запропонувати продовження.

### Suspicious
- **`last_delivery_date <= today - 4 days`**: підписка потрапляє до "ended" якщо остання доставка минула 4+ дні тому. Але якщо остання доставка — завтра — підписка не покаже. Якщо остання доставка була 2 дні тому — теж не покаже. Поріг 4 дні — магічне число без коментаря.
- **`followup_status == 'pending'`** включається в умову "ще не оброблено": але статус `pending` ставиться тільки при CSV-імпорті коли `delivery_number >= 5`. Для звичайних підписок тут буде NULL, не `pending`.

## 13. CSV Імпорт

### Observed behavior
- Парсинг поля "Клієнт": `@handle` → telegram, `0XXXXXXXXX` → телефон, текст → instagram.
- Нормалізація телефонів до формату `+380XXXXXXXXX`.
- `create_subscription_from_import` дозволяє вказати `delivery_number` (1-5): скільки доставок в поточному циклі вже пройшло. Генерує `max(1, 5 - delivery_number)` нових доставок.
- `delivery_number >= 5` → `followup_status = 'pending'` (підписка майже завершена).

### Intended behavior (*hypothesis*)
- Міграція з попередньої системи обліку (таблиця/Excel) в CRM з урахуванням поточного стану підписок.

### Suspicious
- **`delivery_number 5 → 1 delivery + pending`**: `5 - 5 = 0`, але `max(1, 0) = 1`. Тобто навіть "завершена" підписка отримає одну доставку. Можливо це правильно (ще одна остання доставка), але неочевидно.
- **При `delivery_number=1` генерується 4 доставки**: поведінка ідентична звичайному `create_subscription`. Різниця тільки в `is_wedding` та потенційному `followup_status`.
