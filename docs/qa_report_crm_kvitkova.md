QA-звіт: CRM Kvіtkova Povnya — Підготовка до релізу

  🔴 Критичні проблеми (виправити до релізу)

  ---
  1. Модальне вікно блокує кнопку «Зберегти» (клієнти та замовлення)

  Сторінки: /clients, /orders

  Проблема: Після відкриття форми «Додати клієнта» / «Нове замовлення» div[role="dialog"] сам перехоплює pointer-events і не пропускає кліки на кнопку button[type="submit"] всередині. Playwright (і звичайний браузер у певних сценаріях) взагалі
  не може клікнути «Зберегти».

  <div role="dialog" id="clientModal" class="modal fade show" ...>
    ...
    <section class="client-panel"> ← перехоплює події замість кнопки
      <button type="submit">Зберегти</button>

  Чому критично: Якщо форма відкривається і кнопка недоступна, користувач не може зберегти запис. Це може виникати при певних умовах на реальних девайсах / браузерах.

  Виправлення: Перевірити z-index та pointer-events для .client-panel/.order-three-col. Вони не повинні перекривати submit-кнопку.

  ---
  2. Поля дат приймають минулі дати без обмежень

  Підтверджено live-тестуванням:
  - /orders — поле дати доставки: min="null" → можна ввести 2020-01-01
  - /subscriptions — поле first_delivery_date: min="null" → аналогічно

  Проблема: Менеджер може створити замовлення на вчорашній день або в минулому році. Кур'єр отримає маршрут із вже простроченими доставками.

  Виправлення:
  <!-- HTML: -->
  <input type="date" name="delivery_date" min="{{ today }}" required>

  # Python (додати в route):
  from datetime import date
  if delivery_date < date.today():
      return {'error': 'Дата доставки не може бути в минулому'}, 400

  3. Два поля форми клієнта мають name="null" — дані НЕ зберігаються

  Підтверджено live:
  Input: name="null" type="text" placeholder="username"      ← Instagram
  Input: name="null" type="text" placeholder="@username"     ← Telegram

  Проблема: При сабміті форми ці поля передаються як null=значення → бекенд їх ігнорує. Соціальні мережі клієнта ніколи не збережуться.

  Виправлення: Призначити правильні name атрибути: instagram_username, telegram_username.

  4. Немає можливості змінити пароль адміністратора

  Підтверджено live: /settings — Password inputs: 0. Форми зміни пароля в UI немає взагалі.

  Проблема: Якщо пароль AdminPass123 скомпрометований — немає способу його змінити через інтерфейс. Критично перед продакшеном.

  Виправлення: Додати форму зміни пароля в /settings з полями old_password, new_password, confirm_password.

  5. Кредити клієнта не списуються при доставці (фінансова втрата)

  З аналізу коду (/app/blueprints/orders/routes.py, BUG-01):

  # При позначенні доставки виконаною:
  delivery.status = 'Доставлено'
  delivery.delivered_at = datetime.now()
  # ❌ Відсутнє: client.credits -= delivery.locked_price

  Проблема: Клієнти отримують квіти, але їхній баланс не зменшується → пряма фінансова втрата.

  ---
  6. Хардкод credentials бази даних у коді

  З аналізу коду (/app/config.py):
  DATABASE_URL = 'postgresql://kvitkova_user:kvitkova_password@localhost:5432/kvitkova_crm'
  SECRET_KEY = os.environ.get('SECRET_KEY') or 'your-secret-key-here'

  Проблема: Якщо репозиторій стане публічним або буде злитий — credentials відразу у зловмисників. Fallback 'your-secret-key-here' робить Flask sessions тривіально підробними.

  Виправлення:
  DATABASE_URL = os.environ['DATABASE_URL']  # без fallback
  SECRET_KEY = os.environ['SECRET_KEY']      # без fallback

  10. Відсутній CSRF-захист на POST-запитах

  З аналізу коду: Більшість <form method="POST"> не містять CSRF-токен.

  Проблема: Атакуючий сайт може відправити форму від імені залогіненого адміністратора (наприклад, видалити клієнта чи створити замовлення).

  Виправлення:
  from flask_wtf.csrf import CSRFProtect
  csrf = CSRFProtect(app)

  14. Налаштування сесійної cookie для production

  Підтверджено live:
  session cookie: httpOnly=true ✅, secure=false 🟡, sameSite=Lax ✅

  Проблема: secure=False прийнятно для HTTP-розробки, але в production на HTTPS-домені сесія повинна передаватись тільки через HTTPS.

  Виправлення:
  SESSION_COOKIE_SECURE = True  # тільки в production
