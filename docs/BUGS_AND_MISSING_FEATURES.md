## Баги (код робить інакше ніж потрібно)

### BUG-01 — Credits не зменшуються після виконання доставки
- **Де:** `delivery_service.py` → `set_delivery_status`
- **Що є:** при переході в `Доставлено` нічого не списується
- **Що має бути:** після успішної доставки з `client.credits` списується `locked_price` замовлення

### BUG-02 — Debit-транзакція не зменшує `client.credits`
- **Де:** `blueprints/transactions/routes.py:92`
- **Що є:** тільки `client.credits += amount` при credit; debit ігнорує credits
- **Що має бути:** debit типу "списання з клієнта" має зменшувати `client.credits`

### BUG-03 — `delivery_date` не валідується на минуле
- **Де:** `order_service.py:27`, `subscription_service.py:94`
- **Що є:** дата парситься і зберігається без перевірки
- **Що має бути:** дата не може бути раніше сьогодні — валідаційна помилка

### BUG-04 — `time_from > time_to` не блокується
- **Де:** `order_service.py`, `subscription_service.py` — валідація відсутня
- **Що є:** будь-який часовий діапазон приймається
- **Що має бути:** якщо обидва вказані — `time_from` < `time_to`, інакше помилка

### BUG-05 — Телефон отримувача не валідується
- **Де:** `order_service.py:33` — береться як є з форми
- **Що є:** будь-який рядок приймається як телефон
- **Що має бути:** формат `+380XXXXXXXXX`, валідаційна помилка якщо не відповідає

### BUG-06 — Замовлення можна редагувати після `Доставлено`
- **Де:** `order_service.py:143` → `update_order` — немає guard
- **Що є:** редагування дозволено в будь-якому стані
- **Що має бути:** якщо всі доставки мають статус `Доставлено` — редагування заблоковано

### BUG-07 — Підписку з виконаними доставками можна видалити
- **Де:** `subscription_service.py:362` → `delete_subscription`
- **Що є:** видаляє підписку і всі ордери без перевірки
- **Що має бути:** якщо є хоча б одна доставка зі статусом `Доставлено` — видалення заблоковано

### BUG-08 — `followup_status='extended'` можна виставити без реального продовження
- **Де:** `blueprints/dashboard/routes.py:199`
- **Що є:** POST до `/dashboard/subscriptions/<id>/status` з `status=extended` ставить прапор без створення ордерів
- **Що має бути:** статус `extended` виставляється тільки через `extend_subscription` яка реально створює ордери

### BUG-09 — Доставку `is_pickup` або `nova_poshta` можна додати до маршруту
- **Де:** `blueprints/routes/routes.py` — немає перевірки при збереженні маршруту
- **Що є:** будь-яка доставка може потрапити до маршруту
- **Що має бути:** самовивіз і Нова Пошта не можуть бути в маршруті кур'єра

### BUG-10 — Звіти фільтруються по `Order.created_at`, не по `Delivery.delivery_date`
- **Де:** `services/reports_service.py:182-187`
- **Що є:** місячна фільтрація за датою створення замовлення
- **Що має бути:** фільтрація за датою доставки (`Delivery.delivery_date`)

### BUG-11 — Маркетинг-звіт рахує замовлення, а не клієнтів
- **Де:** `services/reports_service.py:176` — `func.count(Order.id)`
- **Що є:** кількість замовлень по маркетинговому джерелу
- **Що має бути:** кількість унікальних клієнтів по маркетинговому джерелу

### BUG-12 — `Transaction.amount` — Integer, але потрібно дробове
- **Де:** `models/transaction.py:11`
- **Що є:** `db.Column(db.Integer, ...)`
- **Що має бути:** `Numeric(10, 2)` — дробові суми (наприклад, 99.50 грн)

### BUG-13 — Dashboard не показує підписки з останньою доставкою завтра
- **Де:** `blueprints/dashboard/routes.py:129`
- **Що є:** фільтр `last_delivery_date <= today - 4` — підписки з майбутніми доставками не потрапляють
- **Що має бути:** підписки з останньою доставкою ≤ сьогодні + buffer показуються в списку ended

## Відсутні фічі (логіки немає взагалі)

### FEAT-01 — Клієнт без Instagram
- **Що є:** `Client.instagram` — `nullable=False`, пошук тільки по instagram
- **Що має бути:** клієнт може мати тільки телефон або тільки telegram. Пріоритет відображення: Instagram → Telegram → телефон. Унікальність — окремо по кожному непустому полю.

### FEAT-02 — `personal_discount` застосовується до замовлення
- **Що є:** `personal_discount` — рядок, ніде не використовується в бізнес-логіці
- **Що має бути:** відсоток знижки застосовується до суми замовлення. Значення фіксується в момент створення замовлення (не на дату доставки).

### FEAT-03 — Фіксація ціни замовлення (`locked_price`)
- **Що є:** в `Order` немає поля для збереженої ціни
- **Що має бути:** при створенні замовлення фіксується `locked_price = base_price * (1 - personal_discount%)`. Саме ця сума списується з credits після доставки.

### FEAT-04 — Два типи debit-транзакцій
- **Що є:** один тип `debit` для всього
- **Що має бути:** розділити на:
  - `debit_client` — списання з балансу клієнта (зменшує `client.credits`)
  - `debit_expense` — витрата магазину (не торкається credits клієнта)

### FEAT-05 — Статус маршруту "Прийнято кур'єром"
- **Що є:** після прийняття через Telegram статус маршруту залишається `sent`
- **Що має бути:** окремий статус `accepted` коли кур'єр прийняв маршрут через Telegram

### FEAT-06 — Архівний статус підписки
- **Що є:** підписка існує тільки в стані `active / completed / cancelled`
- **Що має бути:** після `followup_status = 'declined'` підписка переходить в архів (відображається окремо, не в активних)
