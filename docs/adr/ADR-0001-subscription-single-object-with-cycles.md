# ADR-0001 — Підписка: один об'єкт з циклами

**Статус:** Прийнято  
**Дата:** 2026-05-05

## Контекст

При продовженні підписки система могла або (A) додавати нові замовлення до існуючої підписки, або (B) створювати нову підписку і посилатись на попередню через `parent_subscription_id`.

## Рішення

**Варіант A**: одна підписка, необмежена кількість циклів.

Кожне замовлення містить `cycle_number` (який це круг: 1, 2, 3...) і `sequence_number` (порядковий номер в рамках підписки).

## Причини

- Власник не потребує окремих об'єктів для кожного циклу
- Якщо клієнт пропадає і повертається — нічого не ламається, просто додається новий цикл пізніше
- Простіша модель: вся історія підписки в одному об'єкті
- Знижка і "без вази" при продовженні — вручну; `cycle_number > 1` дає сигнал флористу

## Наслідки

- `extend_subscription()` — єдина точка входу для продовження
- `is_extended` більше не використовується для regular-підписок (лише для import reminders)
- Дашборд виявляє підписки що потребують продовження через date-based SQL (не через `is_extended`)
- Не пропонувати Variant B (нова підписка при продовженні) — рішення закрите
